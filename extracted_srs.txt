# 1\. Introduction

## 1\.1 Project Overview

BuildVerse is a full\-stack, web\-based event management platform designed specifically for prototype\-building competitions\. The platform serves two distinct user roles: Administrators and Participants \(Teams\)\. It digitises the entire event lifecycle — from team registration and QR\-based check\-in, to component inventory management, point tracking, and real\-time leaderboards\.

## 1\.2 Purpose of This Document

This Software Requirements Specification \(SRS\) defines the complete functional and non\-functional requirements for the BuildVerse platform\. It is intended for developers, designers, event organisers, and other stakeholders involved in building or evaluating the system\.

## 1\.3 Scope

The platform covers the following areas:

- Team registration and QR code generation
- Admin\-controlled event day check\-in via QR scan
- Component catalogue with point\-based pricing
- Team shopping cart and component request workflow
- Admin\-controlled component delivery confirmation
- Team dashboard with point balance and purchase history
- Public activity feed showing what teams are buying

# 2\. Stakeholders & User Roles

## 2\.1 Admin Role

Admins are event organisers or volunteers who manage the backend operations of the event\. There may be one or more admins operating simultaneously\.

### 2\.1\.1 Admin Capabilities

- Register new teams manually or review pre\-registered teams
- Generate unique QR codes per team and distribute them \(via email/download\)
- Scan or enter QR codes on the event day to check in teams
- View full team list with check\-in status
- Manage the component catalogue \(add, edit, remove components and point costs\)
- View all pending component requests across all teams
- Mark individual component requests as Delivered once teams collect them
- View all team point balances and transaction histories
- Monitor the real\-time activity feed

## 2\.2 Team / User Role

Teams are participant groups that register for the BuildVerse event\. Each team gets one account/login that all members share or a designated team leader manages\.

### 2\.2\.1 Team Capabilities

- View their own dashboard with available points and point usage summary
- Browse the component catalogue with names, descriptions, and point costs
- Add components to their cart
- View their cart \(pending, delivered, and history\)
- View a public activity feed showing other teams' recent purchases and point usage
- Receive their QR code for event day check\-in

# 5\. System Architecture

## 5\.1 Architecture Style

BuildVerse adopts a Three\-Tier Client\-Server architecture with a RESTful API backend, a Single Page Application \(SPA\) frontend, and a relational database\. The architecture prioritises simplicity, rapid development, and maintainability for an event\-scale deployment\.

# 7\. Key User Flows

## 7\.1 Team Registration & QR Distribution Flow

1. Team visits /register and fills out registration form
2. System creates team account, assigns default point balance
3. System auto\-generates unique QR code \(UUID\-encoded\) and stores it
4. Admin receives notification of new registration
5. Admin navigates to QR Management, selects team\(s\), clicks Send QR via Email
6. System sends email with QR code attachment to team's registered email
7. Team can also view and download QR code from their own dashboard under My QR Code

## 7\.2 Event Day Check\-In Flow

1. Admin opens /admin/checkin on a phone or tablet
2. Admin clicks Start Scanner — device camera activates
3. Team presents their QR code \(printed or on phone screen\)
4. Scanner decodes QR, sends UUID to /api/teams/:id/checkin
5. System validates UUID, checks for duplicates
6. Success: team status updated to Checked In with timestamp; green confirmation shown
7. Duplicate: yellow warning shown — Team already checked in at \[time\]
8. Invalid: red error shown — QR code not recognised

## 7\.3 Component Request & Delivery Flow

1. Team browses catalogue at /catalogue, clicks Add to Cart on desired component
2. System checks stock and point balance; adds item to cart if valid
3. Team navigates to /cart, reviews items and total point cost
4. Team clicks Submit Request — system deducts points and creates order records
5. Activity feed entry is created \(visible to all teams and admin\)
6. Admin sees new pending order in /admin/orders queue
7. Volunteer physically collects the component from the storage area
8. Admin clicks Mark as Delivered next to the order
9. Team's order status updates to Delivered; component appears in My Components

# 8\. Security Considerations

__Threat__

__Mitigation__

Unauthorised access to admin pages

JWT role claim validation on every protected API route; frontend route guards

Teams viewing other teams' point balances

Points balance only returned for authenticated team's own ID; feed shows only spend events, not totals

QR code forgery

QR encodes UUID that is validated server\-side against the database; no client\-trusting data

SQL Injection

Prisma ORM uses parameterised queries by default

XSS attacks

React's JSX escapes output; Content\-Security\-Policy headers set

CSRF

JWT in Authorization header \(not cookies\) eliminates CSRF risk

Brute\-force login

Rate limiting \(express\-rate\-limit\): max 10 attempts per 15 minutes per IP

Point manipulation

Points are only modified server\-side through controlled endpoints; client cannot send point delta directly
